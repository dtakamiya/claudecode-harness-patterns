# Claude Code Incident Response Harness

## 概要

一般的な本番サービス障害を、明示的な指揮系統、読み取り専用調査、人間承認、single-writer、構造化された監査証跡で安全に収束させるハーネスです。速度を優先しつつも、AIが自律的に本番変更を行うことは許可しません。

## 向いているケース

- サービス停止、エラー率上昇、遅延、容量逼迫など、進行中の本番障害
- 複数の証拠を集め、仮説と緩和策を短いサイクルで更新する必要がある対応
- 本番操作に明示承認、rollback、監査証跡が必要な対応
- セッション交代や担当者交代に耐える状態記録が必要な対応

## 対象外

- セキュリティインシデント、侵害調査、フォレンジック（専用手順へエスカレーションする）
- 平時の機能開発、計画的なリリース、恒久対策の実装
- 特定の監視、チャット、チケット、クラウド製品への依存

## 実行原則

1. Incident Commanderが重要度、影響、指揮系統、連絡先を確定する。
2. 調査は読み取り専用で開始し、事実・推測・判断を分離して記録する。
3. 本番変更は、影響、検証、停止条件、rollbackを提示して人間の明示承認を得る。
4. 承認された単一のExecutorだけが、承認範囲内の操作を一つずつ実行する。
5. 操作直後に利用者視点とシステム指標を検証し、悪化時は停止またはrollbackする。
6. 復旧後も観測窓を置き、handoffと事後分析へ証拠を引き継ぐ。

## 最小導入手順

1. 状態雛形をコピーし、incident ID、session ID、重要度、影響、指揮者を記入する。
2. 本番用credentialを調査用と実行用に分離し、既定をread-onlyにする。
3. 本番変更ツールを人間承認必須にし、同時に一つのExecutorだけへ付与する。
4. Hooksを補助的なdeny、記録、通知に使う。権限制御そのものはpermissionsと実行基盤で強制する。
5. 各操作にtimeout、停止条件、rollback、次回確認時刻を設定する。
