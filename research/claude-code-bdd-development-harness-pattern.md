# Claude Code BDD開発ハーネス設計の調査

- 調査日: 2026-07-19
- 対象: Gherkin/BDDおよびATDDをClaude Codeなどのコーディングエージェントで駆動する開発プロセス
- 目的: BDDをハーネス化する際の再利用可能な設計要素と失敗モードを抽出し、本リポジトリで独立パターンとすべきかを判断する
- 関連資料: [claude-code-development-harness-patterns.md](claude-code-development-harness-patterns.md)（公式ドキュメント中心の網羅調査）、[community-harness-implementations-2026.md](community-harness-implementations-2026.md)（OSS実装事例）

## 結論

BDDをコーディングエージェントに載せる価値は、「Gherkinという記法を使うこと」ではなく、**実装前に人間が承認した観測可能な受け入れ条件を固定し、それを完了判定の正本にする**点にある。エージェントは自分が書いたテストを自分で緩める方向に動けるため、受け入れ条件を実装より前に、かつ実装詳細を含まない語彙で固定することが本質的な制御になる。

一方で、BDD特有のコストは残る。グルーコード（step definition）は規模とともに劣化し、UIに結合したシナリオは脆く、AIが生成するGherkinは放置すると曖昧なThenステップ、UI手順の羅列、複数振る舞いの混在に流れる。Specification by Exampleを実際にAI駆動で試した報告では、速度面の優位が場当たり的なアプローチに対して明確でなく、BDDツールの知識がないと設計が劣化するとされている。したがって「BDDを使えば品質が上がる」という前提でハーネスを組むべきではない。

本リポジトリへの推奨は、**BDDを独立パターンとして新設するのではなく、Development Harnessの受け入れ条件定義とTDDサイクルに差し込む「BDDモード」として扱う**ことである。独立パターン化を正当化できるのは、feature → 受け入れ条件 → 仕様 → 計画という多層の仕様スタックと、その各層に人間ゲートを置く運用を本当に必要とする規模の場合に限られる。Lightweight FeatureやMicro Bugfixの規模でGherkin層を挟むのは、得られる保証に対して手数が過大である。

## 調査方法

Web検索とページ取得により、2026年時点で公開されているBDD×AIエージェントの実装・ガイドライン・実践報告を収集した。特に、実際に動いているClaude Code向け実装（ATDDプラグイン）、エージェント向けに整備されたGherkinルール、および否定的な実践報告を意図的に含めた。学術的な効果測定は本調査の範囲では十分に得られておらず、以下の記述は主に実務報告に基づく。

## 1. BDDをAIエージェントで駆動する主要アプローチ

### 1.1 Specification by Example 型

Gherkin形式で人間可読な例を書き、AIに失敗するテストを実装させ、次にテストを通す製品コードを書かせ、最後にリファクタリングする4ステップ。TDDの赤緑リファクタを、受け入れ条件のレイヤーに持ち上げた形である。

利点として、AIに与える作業単位（「良いチャンク」）が明確になること、出力が自動テストで検証可能になること、システムの実挙動と一致することが保証された生きたドキュメントが残ることが挙げられている。

### 1.2 ATDD 型（受け入れテストとユニットテストの2ストリーム）

Claude Code向けのATDDプラグイン実装では、次の7ステップが定義されている。

1. Given/When/Then仕様の作成（ドメイン語彙のみ、実装詳細を含めない）
2. テストパイプラインの生成（パーサー → 中間表現 → テストジェネレータ）
3. 受け入れテストの実行（赤で失敗することを確認）
4. TDDによる実装（ユニットテスト＋コード）
5. 仕様レビュー（実装詳細の漏洩チェック）
6. ミューテーション検査（テストが実際にバグを捕捉するかの検証）
7. 次機能へ反復

重要なのは、実装期に**受け入れテスト（外部の振る舞い＝WHAT）とユニットテスト（内部構造＝HOW）を並行して走らせ、両方が同時に緑になることを完了条件にする**点である。片方だけの合格は完了とみなさない。

### 1.3 Spec-Driven Development (SDD) 型との差異

2026年時点で、GitHub Spec Kit、AWS Kiro、OpenSpec、BMADなど各社がSDDの実装を出しており、SDDとBDDは混同されやすい。区別としては、BDDが「振る舞いをビジネス言語で先に書く」のに対し、SDDは「振る舞い・アーキテクチャ・エッジケース・制約を含む仕様全体を先に書き、エージェントにコード・テスト・ドキュメントを生成させる」とされる。

SDDが解こうとしている失敗モードは intent drift（「ログインを追加して」のような過少指定に対し、モデルが妥当だがチームの意図と一致しない既定値を選ぶ）である。この問題意識はBDDの受け入れ条件固定と同じ方向を向いており、ハーネス設計上は「仕様を正本にする」という一点で共通に扱える。

#### ATDDとSDDの対比

両者は競合する代替案ではなく、カバー範囲が異なる。

| 観点 | ATDD | SDD |
|---|---|---|
| 固定するもの | 外部から観測可能な振る舞いのみ | 振る舞い＋アーキテクチャ＋制約＋エッジケース |
| 記述の語彙 | ドメイン言語（実装詳細を排除） | 設計語彙を含む |
| 正本の形 | 実行可能なテスト | ドキュメント（実行可能とは限らない） |
| 検証手段 | テストが緑になるかで自動判定 | レビュー・チェックリスト（人間またはLLM判定に寄る） |
| 主に防ぐ失敗 | エージェントが完了を偽る、テストを緩める | intent drift |

それぞれの弱点は相補的である。SDDは**検証がドキュメントレビューに留まりやすく**、エージェントが仕様を守ったかを機械的に確認できなければ「より詳しいプロンプト」以上のものにならない。ATDDは**表現力が狭く**、テストにできない非機能要件やアーキテクチャ判断が仕様スタックから抜け落ちる。

選び分けの目安は次のとおり。

- **既存コードベースへの機能追加**: ATDD寄り。受け入れ条件が実行可能なテストになるため完了判定が決定論的になり、エージェント運用で最も効果が出る領域である。
- **新規プロジェクト、または大きな設計判断を含む変更**: SDD寄り。ATDDの受け入れ条件では非機能要件やアーキテクチャを表現できず、そこが空白になる。
- **併用が現実解**: SDDで仕様全体を先に固め、そのうち観測可能な部分をATDDの受け入れテストへ落とす。1.2で挙げたATDD実装の仕様スタック（feature → acs → spec → plan）は実質この併用形である。

ハーネス設計としては、**SDDで書いた仕様のうちどれだけを実行可能な受け入れテストへ変換できたかを品質指標として扱う**のが妥当である。

なお、この対比は本調査で収集した実務報告とツール実装からの整理であり、両手法を対照した効果測定データは得られていない。特にSDDは2026年時点で各社の実装が乱立しており、「SDD」という語が指す内容がツールごとに相当異なる点に注意が必要である。

### 1.4 グルーコード省略型

LLMがGherkinを直接読んで解釈・実行できるため、従来BDD導入の障壁だったグルーコードを書かずに済む可能性が指摘されている。ただしこれは、テスト実行がエージェントの解釈に依存することを意味し、CIでの決定論的な再実行性とトレードオフになる。回帰ゲートとして使うなら、生成されたテストコードを成果物として固定する必要がある。

## 2. ハーネス設計に落とせる要素

### 2.1 仕様スタックと人間ゲート

ATDD実装では feature.md → acs.md → spec.md → plan.md という層を積み上げ、**各層を次へ進む前に人間が承認する**。これは本リポジトリの [human-gate-policy.md](../patterns/human-gate-policy.md) のDecision Packetと同じ構造であり、そのまま接続できる。BDD固有の追加点は、承認対象が「計画」ではなく「観測可能な受け入れ条件」である点である。

### 2.2 Gherkin品質ルールのrules/skill化

エージェント向けに整備された `gherkin-guidelines.md` が公開されており、プロジェクトに置いて参照させる、常時遵守ルールとして与える、skillやsub-agentに変換する、という3つの組み込み方が想定されている。主要ルールは以下。

- 基本原則: 振る舞いを記述する / 人間向けに書く / 例による仕様 / 1シナリオ1振る舞い / 単独実行可能
- シナリオ: 宣言的スタイル、ドメインレベルの抽象化、UI実装詳細を書かない、具体的で現実的なデータ、10ステップ未満
- ステップ: 三人称現在形、Given/When/ThenはArrange/Act/Assert、順序厳守、Or禁止
- 検証: 観測可能で検証可能な結果のみ、曖昧な表現の禁止
- アンチパターン: 複数振る舞いの混在、UI手順の埋め込み、肥大化した準備、曖昧な検証、プレースホルダデータ、Scenario Outlineの濫用

これは本リポジトリの `.claude/rules/` 相当に置くのが最も素直な組み込み先である。

### 2.3 二重グリーン条件

完了判定を「受け入れテスト緑 かつ ユニットテスト緑」とする。エージェントが自分の書いたテストを緩めて完了を主張する failure mode に対する直接の対策になる。

### 2.4 決定論的検証スクリプトによるチェックポイント

ATDD実装では、チェックポイント進行時に決定論的なスクリプト（前チェックポイントの完了確認、ブランチ衛生、設計適合性、差分ミューテーション検査、重複コード検出）が実行される。**判定をLLMに委ねず、スクリプトの終了コードで進行を制御する**という方針は、本リポジトリの既存パターンの `scripts/` および Hooks の考え方と一致する。

### 2.5 ミューテーション検査

受け入れテストが実在するバグを捕捉できるかを検証する工程。AIが「通るテスト」を書きがちである以上、テストの有効性そのものを測る層は理屈として妥当である。ただし実行コストが高く、差分に限定するなどの制約が前提になる。

### 2.6 Claude Code機構への対応

BDDは公式機能ではないため、各要素をどの機構に載せるかを対応付けておく。

| BDD要素 | 載せる機構 |
|---|---|
| Gherkin品質ルール | `.claude/rules/` の常時ルール、または必要時に読むSkill |
| 受け入れ条件のレビュー（実装詳細の漏洩チェック） | 独立コンテキストのSubagent |
| 二重グリーン・チェックポイント判定 | 検証スクリプト＋Hooks（best-effortの多層防御として） |
| 受け入れ条件の承認 | 人間ゲート（Decision Packet） |
| featureファイル・仕様スタック | Gitで版管理する成果物 |

## 3. 観測された失敗モードと限定条件

- **AI生成Gherkinの品質劣化**: 明示的なルールがない場合、曖昧なThenステップ、UI手順中心の記述、複数の振る舞いを1シナリオに混在させる、埋め草のようなプレースホルダ例に流れる。ルールを与えれば改善するとされるが、これは出典が自身のガイドラインを提案する文脈での主張であり、効果の定量的裏付けは示されていない。
- **「正しい理由で失敗しているか」をAIが判別しにくい**: 赤フェーズの妥当性判定はBDDサイクルの要だが、ここがAIの弱点だと報告されている。赤の確認を人間またはスクリプトに残す必要がある。
- **グルーコードの規模劣化**: step definitionは各ステップに対応するコードを要求し、管理しないと1〜2シナリオでしか使われない微妙に異なる定義が大量に生まれる。
- **UI結合による脆さ**: UI要素に依存したシナリオは、ボタン1つの移動で多数のシナリオが同時に壊れる。
- **flakyの不可避性**: テスト数・環境・インフラが増えるほど、確率的に散発的失敗が避けられなくなる。
- **訓練分布への過適合**: 生成されたシナリオがドメイン固有のエッジケースではなく、LLMの訓練分布に沿った一般的パターンを反映するリスクがある。
- **速度優位が確認されない事例**: Specification by ExampleをAI駆動で試した報告では、場当たり的なアプローチに対して速度が実質的に改善しなかったとされ、BDD固有コードの維持とデバッグ層の overhead が便益を正当化しない可能性が指摘されている。
- **組織的な失敗の再演**: 仕様をテスターだけが書くなら、従来のBDDが失敗した構図をそのまま繰り返すという指摘がある。

これらはいずれも個別の実務報告およびブログ記事に基づくものであり、対照実験による効果測定ではない。特に速度に関する否定的報告は単一著者の経験であり、チーム・ドメイン・ツール習熟度を越えて一般化できない。

## 4. 本リポジトリの既存パターンとの対応

| 既存パターン | BDD要素との関係 |
|---|---|
| Development Harness | 要件定義・受け入れ条件・独立レビュー・品質ゲートを既に持つ。BDDは「受け入れ条件をGherkinで書き、実行可能にする」差分にあたり、TDDサイクルの外側に受け入れテスト層を1枚足す形で吸収できる |
| Lightweight Feature Harness | 単一セッションのTDDが前提。Gherkin層と仕様スタックを挟むと手数が過大になり、適合しない |
| Micro Bugfix Harness | 回帰テストで足りるため、BDD層は不要 |
| Incident Response Harness | 対象外 |
| human-gate-policy.md | 仕様スタック各層の承認をDecision Packetとしてそのまま表現できる |
| change-intent-record.md | 受け入れ条件は設計意図の一部であり、featureファイルへの参照をChange Intent Recordのリンク先にできる |

## 5. 設計判断

1. **独立パターンとして新設しない。** Development Harnessに「BDDモード」として組み込む。理由は、実行フロー・ゲート・レビュー体制の大半がDevelopment Harnessと重複し、差分がGherkin層と受け入れテスト実行に限られるためである。
2. **Gherkin品質ルールは `.claude/rules/` に置く。** シナリオ生成とレビューの両方で常時参照させる。
3. **完了条件は二重グリーン**（受け入れテストとユニットテストの同時合格）とし、片方のみの合格を完了とみなさない。
4. **赤フェーズの妥当性確認をAIに委ねない。** 「正しい理由で失敗しているか」の判定は人間ゲートまたは決定論的スクリプトに残す。
5. **受け入れ条件は実装詳細を含まないドメイン語彙で書き、UI要素に結合させない。** 実装詳細の漏洩チェックを独立Subagentのレビュー項目にする。
6. **グルーコード省略型は回帰ゲートに使わない。** CIで再実行する層は生成済みテストコードとして固定する。
7. **ミューテーション検査は差分限定の任意工程とする。** 全体実行はコストに見合わない。
8. **ATDDとSDDは択一にしない。** 既存コードへの機能追加はATDDを既定とし、新規構築や大きな設計判断を含む変更ではSDD的な仕様記述を前段に置く。併用時は「仕様のうち実行可能な受け入れテストへ変換できた割合」を品質指標として扱い、テストに落ちなかった項目は非機能要件・設計判断としてChange Intent RecordまたはADRへ明示的に退避させる。
9. **独立パターン化の再検討条件**: feature → 受け入れ条件 → 仕様 → 計画の多層仕様スタックと層ごとの人間ゲートを実運用で必要とする案件が現れた場合に限り、独立パターンを再検討する。

## 6. 採用チェックリスト

- [ ] その機能の受け入れ条件を、実装前に人間が承認したか
- [ ] 受け入れ条件はドメイン語彙で書かれ、UI要素・実装詳細を含まないか
- [ ] 1シナリオが1つの振る舞いだけを記述しているか
- [ ] Thenステップが観測可能・検証可能な結果を述べているか（曖昧語がないか）
- [ ] 受け入れテストが赤になったとき、正しい理由で失敗していることを人間またはスクリプトが確認したか
- [ ] 受け入れテストとユニットテストが同時に緑か
- [ ] step definitionが1〜2シナリオ専用のものに増殖していないか
- [ ] 変更規模がLightweight Feature相当なら、Gherkin層を挟まない選択を検討したか
- [ ] 新規構築や大きな設計判断を含む場合、受け入れテストで表現できない非機能要件・アーキテクチャ判断を別途記録したか
